# Experiment Log — Cell Segmentation (CS-GY 9223)

This document tracks all experiments, design decisions, bugs found, and results for the cell segmentation pipeline. Visualizations and training curves will be added during final report preparation.

---

## Problem Statement

Given raw DAPI and polyT fluorescence microscopy images from MERFISH experiments on mouse brain tissue, segment cells and assign each decoded mRNA spot to a cell (or background). Evaluated using Adjusted Rand Index (ARI) between predicted and ground-truth spot-to-cell assignments, averaged across 4 held-out test FOVs.

- **Image resolution:** 2048 × 2048 pixels, 0.109 µm/px → ~223 µm × 223 µm per FOV
- **Z-planes:** 5 per FOV (3D data, but initially processed as 2D slices)
- **Training data:** 40 FOVs (FOVs 001–040), using 001–035 for training and 036–040 for validation
- **Test data:** 4 FOVs (FOV_A, FOV_B, FOV_C, FOV_D) — no ground truth provided
- **Ground truth:** Cell boundary polygons in µm coordinates, generated by Cellpose 2.0

---

## Coordinate System (Critical Discovery)

MERFISH uses a non-standard image coordinate convention that caused all early failures:

```
image_row = 2048 - (global_x - fov_x) / pixel_size   ← x-axis is FLIPPED
image_col = (global_y - fov_y) / pixel_size
```

The intuitive mapping (x→col, y→row) places masks in completely wrong image locations. Verified empirically: DAPI signal inside correctly-placed masks is **4.0× higher** than background; with the wrong convention it drops to ~1.0× (indistinguishable from background). This was the primary failure mode for all early runs.

The course team issued an announcement mid-competition noting the same issue with `test_spots.csv` — pre-computed `image_row`/`image_col` columns were added to both `spots_train.csv` and `test_spots.csv` so students could do direct lookup without manual coordinate math.

## Experiment 001 — cyto2 fine-tuning, initial attempt (multiple broken runs)
**SLURM jobs:** 167059, 167060, 167084 (multiple attempts)  
**Status:** Failed — 0 cells detected across all validation FOVs  
**Val ARI:** 0.0000  

### Architecture
- **Base model:** Cellpose cyto2 (pretrained)
- **Input:** `[polyt_z2, dapi_z2]` — 2 channels, single z-plane (z=2 of 5)
- **Training epochs:** 100, chunk size 20
- **LR:** 5e-5, weight_decay=0.1

### Bugs Found & Fixed

**Bug 1 — Wrong validation diameter (`diameter=30`)**  
Cellpose uses the `diameter` parameter to set the scale of its detection filters. Cells in this dataset are ~10 µm, which at 0.109 µm/px corresponds to ~89 pixels. With `diameter=30`, the model's receptive field is tuned for 30px objects and simply cannot detect 89px cells. Fixed to `diameter=88.9`.

**Bug 2 — Wrong coordinate convention in `boundaries_to_mask`**  
The function that converts ground-truth boundary polygons (in µm) to pixel-space masks was using:
```python
col_px = (x - fov_x) / pixel_size   # WRONG
row_px = (y - fov_y) / pixel_size   # WRONG
```
The correct MERFISH convention is:
```python
row_px = (image_size - 1) - (x - fov_x) / pixel_size  # x-axis flipped
col_px = (y - fov_y) / pixel_size
```
With the wrong convention, cell masks were drawn in completely the wrong location — the model was trained with bright DAPI regions labeled as background and dark background regions labeled as cells. Tested all 4 possible axis/flip combinations; the corrected convention gave **4.0× DAPI enrichment** inside mask regions vs ~1.0× for all others. Spot hit rate (fraction of in-cell spots landing inside mask) improved from 20.1% → 28.8%.

**Bug 3 — `weight_decay=0.1` incorrectly blamed**  
Early hypothesis was that `weight_decay=0.1` was destroying pretrained weights (since Adam with 0.1 WD is very aggressive). Changed to `1e-4`. Later confirmed that `0.1` is the Cellpose v4 default (AdamW with decoupled weight decay — different from L2 regularization). Reverted to 0.1. The actual issue was the coordinate bug above.

### Post-fix cell probability diagnosis
After fixing the diameter, fine-tuned model cell probability range was `[-3.67, -1.63]` (entirely negative) while pretrained was `[-9.94, +7.89]`. This confirmed model collapse from training on misaligned masks — model learned to predict "no cells everywhere."

## Experiment 003 — cyto2, max projection + spot density + 300 epochs
**SLURM job:** 167147  
**Status:** In progress (epoch ~160/300 at time of writing, loss ~0.40)  

### Architecture
- **Base model:** Cellpose cyto2
- **Input:** `[polyt_max, dapi_max, spot_density]` — 3 channels
  - `polyt_max` / `dapi_max`: max projection across all 5 z-planes (captures cells at any focal depth)
  - `spot_density`: Gaussian-smoothed mRNA spot count map (σ = 8 px ≈ 0.87 µm), scaled to uint16. Provides direct spatial evidence of cell interiors — mRNA molecules concentrate inside cells.
- **Training epochs:** 300
- **LR:** 1e-5 (reduced from 5e-5 for more careful fine-tuning)
- **weight_decay:** 0.1
- **Diameter at inference:** 0 (auto-detect per FOV)
- **cellprob_threshold:** -1.0 (lower threshold → detects more cells)

### Rationale for each change
- **Max projection:** Fluorescent signal for any given cell may peak at a z-plane other than z=2. Max projection across all 5 planes ensures no cells are missed due to focal depth.
- **Spot density channel:** The decoded mRNA spots are the quantity being assigned to cells — adding their spatial density as an input gives the model direct information about where cells are, independent of image staining quality.
- **300 epochs:** Loss was still declining at epoch 100 in Exp 002. More epochs allow fuller convergence.
- **LR=1e-5:** Lower LR preserves pretrained feature representations while still adapting to tissue-specific patterns.
- **cellprob_threshold=-1.0:** Exp 002 detected 39–86 cells where pretrained found ~114. Lowering threshold recovers marginal cells the model is uncertain about.
- **diameter=0:** Auto-detect lets Cellpose estimate cell size per FOV rather than assuming fixed 88.9 px.

## Pipeline Overview (as of Exp 003/004)

```
Raw .dax files (18 per FOV)
        ↓
load_fov_images() → dapi (5,2048,2048) uint16
                  → polyt (5,2048,2048) uint16
        ↓
Max projection → dapi_max (2048,2048)
              → polyt_max (2048,2048)
        ↓
Spot density map → spot_density (2048,2048) uint16
  (spots_train.csv / test_spots.csv, σ=8px Gaussian)
        ↓
Stack → img (3,2048,2048) [polyt_max, dapi_max, spot_density]
        ↓
Cellpose fine-tuned model
  (cyto2 or cyto3 base, 300 epochs, LR=1e-5)
        ↓
pred_masks (2048,2048) — 0=background, 1..N=cell IDs
        ↓
Spot lookup: cluster_id = pred_masks[image_row, image_col]
        ↓
submission.csv (spot_id, fov, cluster_id)
```

## Key Technical Notes

### Cellpose v4 specifics
- `model_type` argument is ignored in v4.0.1+ (single architecture for all models)
- `train_seg()` returns `(model_path, train_losses, ...)` — must unpack tuple
- Default `weight_decay=0.1` uses AdamW decoupled weight decay (not L2 — 0.1 is valid)
- Default `learning_rate=1e-5` in v4 (was 0.005 in v3 — 500× lower)
- `channel_axis=0` for `(C, H, W)` inputs

### Submission format
- `cluster_id` must be a non-empty string — `"background"` for extracellular spots
- Cluster IDs are arbitrary strings; ARI is label-permutation invariant
- `generate_submission.build_submission()` handles FOV-prefixed IDs automatically

### Evaluation
- ARI is computed per FOV and averaged — all 4 test FOVs weighted equally
- Public leaderboard: 2 FOVs; private: 2 FOVs (final ranking uses private)
- Trivial strategies (all background, all one cell, random) all score ~0.0

**SLURM:** `sbatch ensemble_infer.sbatch`

### Method
`ensemble_infer.py` loads all four trained models (cyto2, cyto3, nuclei, multiscale). For each test spot, each model assigns it to a cell or "background". Majority vote across models determines the final cluster ID. Ties broken by first-listed model.

This bypasses pixel-mask merging (hard, since cell IDs don't align across models) and directly optimises the quantity ARI is computed on.

## Experiment 013 — Non-Cellpose architectures

### 013a — StarDist2D (`stardist`)
**SLURM jobs:** 169805 (initial, cancelled at ep 28), 171438 (resume, cancelled ep 35), 171474 (resume, cancelled ep 35), 171501 (val-only)
**Status:** In progress (chained resumes)

#### Architecture
StarDist2D fine-tuned from pretrained `2D_versatile_fluo`. Polygon-based segmentation via radial distance regression — fundamentally different from Cellpose's flow-field head.

- **Input:** DAPI max-projection only, single channel (pretrained weights expect 1-ch)
- **Normalization:** per-image 1–99.8 percentile
- **Training:** 200 epochs, 100 steps/epoch, built-in StarDist augmentation
- **Validation data:** held-out FOVs 036–040 passed to `model.train(validation_data=…)`
- **Post-training:** `optimize_thresholds()` on val

#### Results
| Training extent | Val ARI | Kaggle |
|---|---|---|
| 28 epochs (partial, 169805 cancelled) — `stardist_v1` | 0.8039 | **0.7627** ✅ (current best) |
| ~98 epochs (post 171438+171474 chain) | 0.8269 | not submitted (weights overwritten by next chain) |
| ~180 epochs (chained resumes) — `stardist_v2` | 0.8247 | **0.7421** ❌ regression |

Despite StarDist having a **lower** val ARI than the best Cellpose variant (0.8039 vs 0.8361), 28-ep scored **higher** on Kaggle (0.7627 vs 0.7464 for cyto2). The radial-polygon head generalizes better across the val→test domain shift (test FOVs ~15% brighter).

**But pushing StarDist further hurts.** At ~180 epochs val rose to 0.8247 (+0.02) but Kaggle dropped to 0.7421 (-0.02). Conclusion: ~28 epochs is the sweet spot for generalization on this dataset. The model is overfitting to val FOVs 036–040, which drift in the same direction as train FOVs but not with the test FOVs' brightness shift.

### 013b — InstanSeg (zero-shot)
`infer_instanseg.py` runs pretrained `fluorescence_nuclei_and_cells` zero-shot on val and test. Explored as a no-training alternative; val ARI below Cellpose baseline, not pursued.

### 013c — U-Net (vanilla PyTorch, scaffolded only)
`src/unet.py`, `train_unet.py`, `infer_unet.py`: 3-class semantic (bg / interior / boundary) + watershed. Scaffolded and ready to train, but deprioritized in favor of StarDist training after the 0.7627 result. Not yet run.
